# Plan and ToDos

## Vision

Have a CLI tool to initialize a new Nimbus project.
This sets up the project structure and installs the necessary dependencies.

The Structure will follow a Domain Driven Design (DDD) approach with a Hexagonal Architecture.

The Source of truth for the API will be an AsyncAPI specification in JSON format.

Based on the AsyncAPI specification the Nimbus framework should register the commands, queries and event on the router structure.
Also the router will provide a route for a health check and routes for a schema registry.
The user now only has to implement the business logic and connects it to the route handlers.

**Claude:**
Your vision is excellent and very well thought out. This approach aligns perfectly with your simplicity goals while providing
professional-grade tooling. Let me validate your thinking:

Why AsyncAPI + AJV + Schema Registry is Smart

1. AsyncAPI as Single Source of Truth

-   Defines all messages (commands/queries/events) in one place
-   Documents your API automatically
-   Industry standard (like OpenAPI but for async/event-driven)
-   Enables codegen → less boilerplate, fewer mistakes

2. AJV Validator Design is Correct

-   Pre-register schemas via validator.addSchema()
-   Messages reference schemas via dataschema URL
-   Router validates automatically when dataschema is present
-   If no dataschema → requires explicit allowUnsafeInput: true

This forces intentional decisions: "I'm skipping validation" vs. accidental omission.

3. Schema Registry Future

-   AsyncAPI → Schema Registry is a natural progression
-   Enables schema evolution (versioning, compatibility checks)
-   Supports microservices discovering each other's contracts
-   You're building the foundation now (URL-based schema references)

Validating Your Philosophy

✅ Keep allowUnsafeInput as opt-in (unsafe = explicit flag)

You're right. Best practice = validate by default. The current warning (router.ts:182-185) is good - it reminds developers during
prototyping that they're being unsafe. In production, messages should have schemas.

✅ Keep per-type routers (command/query/event)

I questioned this earlier, but with AsyncAPI in mind, it makes sense:

-   AsyncAPI documents commands, queries, events separately
-   Different message types may have different routing strategies later (commands = 1:1, events = pub/sub)
-   Codegen can generate separate routers based on AsyncAPI channels

✅ Schema registration pattern is right

// Generated by Nimbus CLI from asyncapi.json
validator.addSchema(addRecipeSchema);
validator.addSchema(getRecipeSchema);

// Messages now reference these schemas
const command: Command = {
// ...
dataschema: 'https://nimbus.overlap.at/schemas/commands/add-recipe/v1',
data: { ... }
};

This is exactly how JSON Schema + HTTP APIs work. You're applying proven patterns to message-driven architecture.

Suggested Refinements

1. AsyncAPI Structure

Your AsyncAPI should look something like:
asyncapi: 3.0.0
info:
title: Gustav Recipe API
version: 1.0.0

channels:
recipeCommands:
address: /command
messages:
addRecipe:
$ref: '#/components/messages/AddRecipe'

    recipeQueries:
      address: /query
      messages:
        getRecipe:
          $ref: '#/components/messages/GetRecipe'

components:
messages:
AddRecipe:
name: at.overlap.nimbus.add-recipe
payload:
$ref: '#/components/schemas/Recipe'
GetRecipe:
name: at.overlap.nimbus.get-recipe
payload:
$ref: '#/components/schemas/GetRecipeParams'

    schemas:
      Recipe:
        type: object
        properties:
          slug: { type: string }
          title: { type: string }
          # ...

2. Nimbus CLI Codegen Flow

nimbus generate --asyncapi asyncapi.json --output src/generated/

Should generate:

-   Schema registration (src/generated/schemas.ts):
    import { getValidator } from '@nimbus/core';

export function registerSchemas() {
getValidator().addSchema(addRecipeSchema);
getValidator().addSchema(getRecipeSchema);
// ...
}

-   Type definitions (src/generated/types.ts):
    export type AddRecipeCommand = Command<Recipe> & {
    type: 'at.overlap.nimbus.add-recipe';
    };
-   Router setup (src/generated/routers.ts):
    export const commandRouter = new MessageRouter('command');
    // Handlers registered manually or via convention

3. Schema URL Convention

Standardize schema URLs:
https://nimbus.overlap.at/schemas/{type}/{name}/{version}

Examples:

-   https://nimbus.overlap.at/schemas/commands/add-recipe/v1
-   https://nimbus.overlap.at/schemas/queries/get-recipe/v1
-   https://nimbus.overlap.at/schemas/events/recipe-added/v1

This enables:

-   Versioning (v1, v2, etc.)
-   Schema registry lookups
-   Clear naming conventions

4. Validator Schema Registration Helper

Add a convenience method to validator:
// validator.ts
public addSchemas(schemas: SchemaObject[]): void {
for (const schema of schemas) {
this.addSchema(schema);
}
}

Then generated code can do:
registerSchemas() {
getValidator().addSchemas([
addRecipeSchema,
getRecipeSchema,
// ...
]);
}

5. Runtime Schema Validation Reporting

In Gustav, you're using allowUnsafeInput: true everywhere. Once you have AsyncAPI + codegen:
// Before (manual, no schema)
router.register(AddRecipeCommandType, handler, { allowUnsafeInput: true });

// After (generated, with schema)
router.register(AddRecipeCommandType, handler);
// No allowUnsafeInput needed - schema is in the message

Implementation Phases

Phase 1: Current (Manual)

-   Write handlers manually
-   Use allowUnsafeInput: true during dev
-   Validate manually if needed

Phase 2: AsyncAPI Definition

-   Create asyncapi.json for Gustav
-   Define all messages and schemas
-   Keep manual handlers

Phase 3: Nimbus CLI Codegen

-   Build CLI that reads AsyncAPI
-   Generate schema registration code
-   Generate TypeScript types
-   Optionally generate router boilerplate

Phase 4: Schema Registry

-   Host schemas at URLs (Nimbus schema registry service)
-   Support schema versioning
-   Schema compatibility checking
